# 火箭软着陆项目 Agent 工作说明

## 项目目标

本项目研究一维火箭垂直软着陆控制，以 Standard PPO 和 Energy PPO 为主要模型。Agent 的任务是协助检查代码、运行实验、整理结果和完善提交材料，但不得脱离实际运行数据编造实验结论。

## 工作范围

1. 只在 `submission_version` 中整理最终提交内容，不破坏外层原始工程。
2. 修改前先阅读 `README.md`、`AI_USAGE.md`、相关代码和已有结果文件。
3. 保留能够运行和复现实验的代码、模型、配置、图像及原始结果。
4. 未经明确要求，不删除模型、正式结果或原始工程文件。

## 实验规范

1. 主推模型为 Standard PPO 和 Energy PPO。
2. 正式实验包括：
   - Standard PPO 的演示、标准分析和奖励消融；
   - Energy PPO 的演示、标准分析、结构消融及与 Standard PPO 的对比；
   - 与 PID、MPC、ET-MPC 等传统控制方法的对比；
   - 与 SAC、TD3 等其他强化学习方法的对比；
   - 基础奖励失败和正式模型压力失败案例分析。
3. 统一评价使用 11 个标准、泛化和鲁棒性场景，每个正式场景测试 30 回合。
4. 同一对比中的方法必须共享 episode seed；标准 100 回合配对评价使用 seed 2026—2125。
5. 代表轨迹固定使用相同的 episode 0 初始条件，避免因起点不同造成不合理对比。
6. 成功条件为触地速度绝对值不超过 2 m/s；燃料耗尽后实际推力必须严格为 0。

## 数据与结论要求

1. 所有数值必须来自实际运行生成的 JSON、CSV、日志或轨迹文件。
2. 修改报告前必须核对表格、图片和原始结果，不能根据预期补写数据。
3. 基础奖励高空燃料耗尽案例必须标为“基础奖励消融失败”，不能写成 Standard PPO 正式模型失败。
4. 正式 30 回合评价与扩展压力搜索要分开说明，不能把扩展搜索中的临界失败隐藏或混入正式成功率。
5. MPC 和 ET-MPC 的结果只代表当前轻量实现与参数设置，不能扩展为对算法理论能力的否定。

## 文件与代码要求

1. 优先使用统一入口 `run_study.py`，并保证分组运行和 `all` 模式不调用已删除文件。
2. 结果应分别保存到 `results/01_standard_ppo` 至 `results/06_report_alignment` 对应目录。
3. 文件路径使用相对路径，不写入本机用户名、桌面目录或下载目录。
4. 修改生成型 Markdown 时，同时修改对应生成脚本，防止重新运行后恢复旧内容。
5. 修改代码后进行语法检查，并尽可能运行 `smoke_tests.py` 和 `verify_reproducible_outputs.py`。

## 文档写作要求

1. 使用课程实验报告语气，以学生本人身份陈述，不使用“作为 AI”等表述。
2. 文字应准确、清楚、适度详细，避免口语化、夸张表达和无依据结论。
3. README 应明确环境安装、快速检查、分组复现、完整复现和结果查看方式。
4. 最终回复应说明实际修改内容、验证结果以及仍需人工处理的问题。
