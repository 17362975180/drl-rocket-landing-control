# AI 工具使用说明

## 1. 使用的 AI 工具

本项目主要使用了以下两类 Agent 工具：

1. **OpenAI Codex**：用于读取工程文件、检查代码、执行实验与验证命令、整理结果目录以及修改说明文档。
2. **Claude Code + mimo-2.5-pro**：使用 Claude Code 作为代码 Agent 运行界面，接入 mimo-2.5-pro 模型，用于代码分析、方案讨论、程序修改、错误排查和报告文字辅助。

本项目没有把普通聊天记录当作“提示词”。Agent 每次开始处理工程任务时，首先读取项目根目录下的 `AGENTS.md`，再按照其中的项目背景、实验规范和文件操作要求完成任务。

## 2. AI 的实际使用方式

### 2.1 代码与环境检查

- 检查火箭动力学、燃料消耗、质量变化、空气阻力、推力惯性和终止条件；
- 排查燃料耗尽后仍可能产生推力等物理约束问题；
- 检查 Standard PPO 和 Energy PPO 的环境、训练、评价与绘图代码；
- 根据运行报错定位依赖、路径、模型加载和数据格式问题。

### 2.2 实验执行与整理

- 协助运行标准评价、统一 11 场景测试、奖励与结构消融实验；
- 协助完成传统控制算法和其他强化学习算法对比；
- 检查不同方法是否采用相同场景、回合数和配对随机种子；
- 按统一目录保存模型、JSON、CSV、轨迹图、成功率热图和失败案例。

### 2.3 报告与提交工程整理

- 根据本地实验结果整理报告中的表格、图像和文字分析；
- 检查报告结论是否与原始 JSON、CSV 和轨迹数据一致；
- 整理 `submission_version` 的 README、运行入口和结果目录；
- 删除已经失效的旧报告生成流程，保留可复现的正式实验程序。

本人通常以自然语言向 Agent 说明当前目标，例如检查某组实验、统一随机种子、整理提交目录或修改报告。Agent 读取 `AGENTS.md` 后检查相关文件并执行任务，本人再根据运行结果和报告要求确认或继续调整。

## 3. 项目级 Agent 提示词

本项目使用的 Agent 提示词保存在 `AGENTS.md` 中。它属于每次任务开始前读取的长期工程规则，不是某次对话记录。内容如下：

```text
# 火箭软着陆项目 Agent 工作说明

## 项目目标

本项目研究一维火箭垂直软着陆控制，以 Standard PPO 和 Energy PPO 为主要模型。Agent 的任务是协助检查代码、运行实验、整理结果和完善提交材料，但不得脱离实际运行数据编造实验结论。

## 工作范围

1. 只在 submission_version 中整理最终提交内容，不破坏外层原始工程。
2. 修改前先阅读 README.md、AI_USAGE.md、相关代码和已有结果文件。
3. 保留能够运行和复现实验的代码、模型、配置、图像及原始结果。
4. 未经明确要求，不删除模型、正式结果或原始工程文件。

## 实验规范

1. 主推模型为 Standard PPO 和 Energy PPO。
2. 正式实验包括：
   - Standard PPO 的演示、标准分析和奖励消融；
   - Energy PPO 的演示、标准分析、结构消融及与 Standard PPO 的对比；
   - 与 PID、MPC、ET-MPC 等传统控制方法的对比；
   - 与 SAC、TD3 等其他强化学习方法的对比；
   - 基础奖励失败和正式模型压力失败案例分析。
3. 统一评价使用 11 个标准、泛化和鲁棒性场景，每个正式场景测试 30 回合。
4. 同一对比中的方法必须共享 episode seed；标准 100 回合配对评价使用 seed 2026—2125。
5. 代表轨迹固定使用相同的 episode 0 初始条件，避免因起点不同造成不合理对比。
6. 成功条件为触地速度绝对值不超过 2 m/s；燃料耗尽后实际推力必须严格为 0。

## 数据与结论要求

1. 所有数值必须来自实际运行生成的 JSON、CSV、日志或轨迹文件。
2. 修改报告前必须核对表格、图片和原始结果，不能根据预期补写数据。
3. 基础奖励高空燃料耗尽案例必须标为“基础奖励消融失败”，不能写成 Standard PPO 正式模型失败。
4. 正式 30 回合评价与扩展压力搜索要分开说明，不能把扩展搜索中的临界失败隐藏或混入正式成功率。
5. MPC 和 ET-MPC 的结果只代表当前轻量实现与参数设置，不能扩展为对算法理论能力的否定。

## 文件与代码要求

1. 优先使用统一入口 run_study.py，并保证分组运行和 all 模式不调用已删除文件。
2. 结果应分别保存到 results/01_standard_ppo 至 results/06_report_alignment 对应目录。
3. 文件路径使用相对路径，不写入本机用户名、桌面目录或下载目录。
4. 修改生成型 Markdown 时，同时修改对应生成脚本，防止重新运行后恢复旧内容。
5. 修改代码后进行语法检查，并尽可能运行 smoke_tests.py 和 verify_reproducible_outputs.py。

## 文档写作要求

1. 使用课程实验报告语气，以学生本人身份陈述，不使用“作为 AI”等表述。
2. 文字应准确、清楚、适度详细，避免口语化、夸张表达和无依据结论。
3. README 应明确环境安装、快速检查、分组复现、完整复现和结果查看方式。
4. 最终回复应说明实际修改内容、验证结果以及仍需人工处理的问题。
```

## 4. 人工确认与责任划分

AI 生成或修改的内容需要经过人工检查，不能直接作为实验依据。本人主要负责：

- 确定课题目标、主要模型、实验内容和评价标准；
- 决定 Agent 每次需要完成的任务，并检查修改结果；
- 运行或确认训练、评价、消融、对比和失败搜索实验；
- 核对模型、随机种子、场景配置、终止原因和原始结果文件；
- 对报告表格、图像和结论进行最终选择与确认。

## 5. 实验数据来源

报告中的实验数值不是由语言模型凭空生成，而是来自 `results/` 中保存的正式运行结果。例如：

- 标准 100 回合中，Standard PPO 与 Energy PPO 的成功率均为 100.0%；
- Standard PPO 的平均最终速度误差、燃料消耗和着陆时间分别为 1.174 m/s、4.396 kg 和 7.283 s；
- Energy PPO 对应指标分别为 0.287 m/s、3.005 kg 和 4.707 s；
- 统一 11 场景中，Standard PPO 平均成功率为 80.3%，Energy PPO 为 100.0%；
- 基础奖励、高扰动和动作延迟失败案例均保留了原始轨迹及参数。

结果完整性可通过 `python verify_reproducible_outputs.py` 检查，也可以直接核对 `results/` 中的 JSON、CSV、PNG 和轨迹文件。

## 6. 使用说明

Codex 和 Claude Code + mimo-2.5-pro 在本项目中承担工程辅助工作，能够提高代码检查、实验执行和文档整理效率，但其建议仍可能存在代码、物理含义或文字表述错误。最终结论仅以本地实际运行结果和人工确认内容为准。
